Основное
-------
Была выбрана метрика LeaderRank - он определяет влияние узла в сети, то есть как часто при случайном блуждании мы будем попадать в вершину i. 
Это помогает в банковской сфере для отслеживания банковских переводов, для рекомендаций (как часто ссылаются на товар и тп), в распределенных системах (поиск самых критически важных узлов) и тп.
LeaderRank похож на PageRank, но добавляется фиктивная вершина с ребрами от всех до всех вершин - чтобы не делать случайных телепортаций в графе, а заменить их на два перехода по ребрам, смежными с фиктивной вершиной. 
Алгоритм:
- В граф добавляют новую вершину и двунаправленно соединяют со всеми вершинами (аналог телепортации в PageRank)
- Влияние вершины v (LR_v(0) = 1), а LR фиктивной вершины =  0
- После чего данные в момент времени t пересчитываются через LR_i(t+1) = \sum_{j\in in(i)} LR_j(t) / k_out(j) пока не сойдутся. 
- При сходимости, надо удалить фиктивную вершину и распределить влияние между всеми: LR_v = LR_v + LR_f(T)/N

Это просто альтернативная версия PageRank - что является очень известной метрикой повсеместно, а гугл в свое время и смог выстрелить благодаря ней.

Вообще, это задача по-сути Марковской цепи. Поэтому, если бы не было ограничений по памяти, можно было бы использовать метод Гаусса (но он слишком долгий) или просто возвели бы матрицу смежности A в степень n и домножили бы на LR_0 (ans = A^n * LR_0). 
1) Первый способ - просто как раз и реализовать перемножение матриц, которые не вмещаются в память, а после сделать быстрое возведение в степень для ответа. Но тогда мы теряем разреженность матрицы, что дорого по памяти. Да и это все еще O(n^3logk) (O(n^3logk/B/sqrt(M)) во внешней памяти) - что очень много, хотя тут легко использовать многопоточность 
2) Можно все ребра отсортировать во внешней памяти по исходящей вершине с помощью External Merge Sort. Тогда получится список смежности и за линию можем посчитать степень каждой вершины. Можем дописать out_degree каждому ребру. Кроме того, пусть у нас уже есть файл LR(t-1) отсортированными по вершинам. Оба файла отсортированы по одному ключу, поэтому можно бежаться по ним обоим последовательно, используя два указателя, пока вершины совпадают. Для каждого ребра вычисляем d = LR_from(t-1)/(out_degree) и записываем новый файл (to, d). Новый файл нужно отсортировать по to и просуммировать d с одинаковыми вершинами to - получается новый LR, продолжаем новую итерацию пока все не сойдется. По-сути тут только затраты на сортировку во внешней памяти, что весьма неплохо. Кроме того алгоритм подходит под разные размеры графа и тп, нет гиперпараметров. Но такое реализовывать неинтересно - просто сортировку слиянием писать многопоточную. На практике выбрал бы именно такой способ, используя готовое решение для сортировки. O(E/B*log_{M/B}(E/B)) на диск, O(ElogE/k + I*(ElogE+V)/k) на внешнюю память. 
3) (Реализовано) Тк средняя степень вершины 50, сам массив LR для очередного t-1 и out_degree должен вмещаться в RAM (значит можно хранить в RAM все предыдущие значения на прошлой итерации и пересчитывать out_degree без сортировки за линию). Тем не менее, мы хотим писать в несколько потоков и наивный пересчет LR потребует отдельного массива для каждого потока, чтобы не было гонки данных. Вместо этого можно разбить вершины на k батчей одинакового размера. Давайте линейно обработаем все ребра и посчитаем для вершин их степень в оперативной памяти. Смотрим на ребро (from, to) и записываем ребро в батч, где находится вершина to. У нас есть k файлов, где ребра разбиты по to в k батчей. Для очередной итерации уже есть LR_{t-1} надо пересчитать LR_{t}. Запускаем k потоков, каждый берет один свой батч. Каждый поток независимо обрабатывает ребра в батче и считает next_rank - можно отдельно выделить на каждый поток свой массив (все равно они берут интервалы и суммарно массивы будут длины n, а значит поместятся в память), чтобы не было проблем с false sharing. Получается Lock-free алгоритм с линейной сложностью. Батчи можно делить не по номеру вершины, а по сумме степеней вершин, чтобы потоки выполняли задачу примерно одинаковое время – у нас же есть гиперузлы. В итоге O(E) в SSD, O(V) в RAM, O(E/K + V) по времени. 
3.5) Раз у нас есть гиперузлы, ребер в батчах может быть разное количество. Поэтому после вычисления степени вершин (как в п.2) можно было разделить их на тяжелые и легкие. Тяжелых мало, поэтому можно в RAM хранить счетчики LR для этих вершин и быстро пробежать по таким ребрам. А легкие ребра обрабатывать, как раньше.
